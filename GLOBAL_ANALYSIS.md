## 三、一致性问题诊断

### 问题1: 三套独立Web应用，无统一入口

```
当前:                    理想:
用户→:5010 → 系统A      用户→统一门户 → 系统选择器
   →:5012 → 系统B                   → 进入具体系统
   →:5023 → 系统C
```

- 用户需记忆3个不同端口号
- 每个系统独立鉴权（目前都无鉴权）
- 系统间无法互通数据

### 问题2: 四套不同的前端设计

| 系统 | 主题色 | 布局 | 组件库 | 风格 |
|:-----|:-----:|:----:|:------:|:----:|
| 迁移评估 | 深蓝#1A3C6E | 渐变+毛玻璃 | 自制 | 现代 |
| Z-DBMate | 标准蓝#0077CC | 渐变+毛玻璃 | 自制 | 现代 |
| DWS部署 | 靛蓝#283593 | 渐变+毛玻璃 | 自制 | 现代 |
| 投标系统 | 蓝色 | Bootstrap | Bootstrap | 传统 |
| HCCDE题库 | — | Tkinter | 原生 | 桌面传统 |

- 菜单结构不一致：左侧导航项位置不同
- 颜色主题都不统一（虽然各具特色但缺乏品牌一致性）
- 按钮/卡片/表格等组件风格有细微差异

### 问题3: 系统间没有数据流通

- 迁移评估系统评估完成后，无法**一键跳转**到Z-DBMate部署OLTP或DWS部署数仓
- DRS/DataX数据迁移环节没有工具支撑
- 项目级数据（客户名、评估报告、部署记录）无法关联

### 问题4: 缺少企业级特性

| 特性 | 当前 | 行业标准 |
|:-----|:----:|:--------:|
| 用户管理 | ❌ | ✅ SSO/LDAP |
| 操作审计 | ❌ | ✅ 审计日志 |
| 多租户 | ❌ | ✅ 项目隔离 |
| 权限控制 | ❌ | ✅ RBAC |
| 数据持久化 | ⚠️ 部分 | ✅ 数据库 |
| 国际化 | ❌ | ✅ i18n |
| CI/CD集成 | ❌ | ✅ API+Webhook |

### 问题5: HCCDE题库和投标系统完全独立

- HCCDE是离线EXE，与在线工具链无关联
- 投标系统有独立的Web UI，但与其他系统无联动

## 四、优化建议

### 🥇 建议一：统一门户（一键启动）

> **投资**: 低 ｜ **收益**: 高 ｜ **优先级**: P0

在现有架构上增加一个**统一门户页面**，替换三个base.html底部的架构栏：

```
http://127.0.0.1:8080  → 统一门户
     ├─ 🧭 迁移智能评估 → :5010
     ├─ ⚙️ Z-DBMate     → :5012
     ├─ 🏗️ DWS部署      → :5023
     ├─ 📋 投标系统     → :5000
     └─ 📚 HCCDE题库    → 离线EXE下载
```

每个卡片显示：系统状态（运行中/维护中）、版本号、最近使用记录。

### 🥇 建议二：统一前端组件库

> **投资**: 中 ｜ **收益**: 高 ｜ **优先级**: P0

抽取**共享CSS组件库**，所有Web系统统一引用：

```
shared-css/
├── brand.css       ← 颜色/字体/间距统一变量
├── components.css  ← 按钮/卡片/表格/标签/表单
├── layout.css      ← 侧边栏/顶部栏/页面布局
└── icon-font.css   ← 统一图标字体
```

三个系统的base.html统一用 `<link rel="stylesheet" href="/shared-css/brand.css">`。

### 🥈 建议三：系统间集成跳转

> **投资**: 低 ｜ **收益**: 高 ｜ **优先级**: P1

迁移评估系统增加"部署目标环境"按钮，根据源端类型推荐：

```python
# 评估完成 → 推荐目标
if result.target_type == "dws":
    if result.source_type in ("oracle", "mysql", "gp"):
        # 推荐 DWS 数仓部署
        show_button("部署DWS数仓 →", ":5023")
    if result.source_type in ("mysql", "pg"):
        # 推荐 GaussDB OLTP
        show_button("部署GaussDB OLTP →", ":5012")
```

### 🥈 建议四：引入数据库持久化

> **投资**: 中 ｜ **收益**: 高 ｜ **优先级**: P1

当前数据存在评估报告的DOCX文件和内存中，建议引入SQLite：

```
data/
├── projects.db     ← 项目信息、客户信息
├── assessments.db  ← 评估记录、评分历史
└── deployments.db  ← 部署记录、部署日志
```

一个客户一条记录，关联评估报告+部署记录。

### 🥈 建议五：构建能力矩阵

> **投资**: 低 ｜ **收益**: 中 ｜ **优先级**: P1

在统一门户中展示完整能力矩阵，覆盖数据库全生命周期：

| 阶段 | 产品 | 对标UGO | 对标AWS |
|:----|:-----|:-------:|:-------:|
| **评估** | 迁移评估系统 | ✅ 兼容性分析 | ✅ DMS Schema Conversion |
| **规划** | 迁移评估系统 | ✅ 工作量估算 | ✅ DMS Fleet Advisor |
| **部署** | Z-DBMate+DWS系统 | ❌ | ✅ DMS + RDS |
| **迁移** | DRS/DataX(外部) | ✅ DRS | ✅ DMS |
| **验证** | DWS验证系统 | ✅ SQL审核(基础) | ✅ DMS Data Validation |
| **考试** | HCCDE题库 | ❌ | ❌ |

### 🥉 建议六：HCCDE题库Web化

> **投资**: 高 ｜ **收益**: 中 ｜ **优先级**: P2

将HCCDE题库从Tkinter迁移到Web，放入统一门户。题库本身就是GaussDB知识库，与迁移评估系统的DWS规则相互补充。考试中的"不兼容项"知识点可作为规则来源。
