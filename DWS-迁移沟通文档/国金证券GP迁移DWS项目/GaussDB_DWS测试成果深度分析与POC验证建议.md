## 一、项目背景分析

### 1.1 迁移动因

| 驱动因素 | 具体内容 |
|----------|----------|
| **信创政策要求** | 集团要求近几年内完成系统信创改造 |
| **硬件过保** | 生产环境服务器将于2024年过保，新采购仅限国产服务器 |
| **时间紧迫** | 需在服务器过保前完成所有模块和组件的信创改造 |

### 1.2 选型要求

1. **兼容性**：信创数据库需满足当前数据平台绝大多数使用场景
2. **迁移能力**：具备从GP迁移的工具链（模型迁移工具 + 数据迁移工具），降低迁移成本
3. **复用性**：尽可能复用旧系统的调度系统、ETL模块、数据仓库模型
4. **前瞻性**：兼顾产品创新性和先进性，满足后续升级改造需求

## 四、GP与DWS语法差异对照表（迁移必知）

| 差异点 | Greenplum | GaussDB(DWS) | 影响评估 |
|--------|-----------|--------------|----------|
| 行列存方式 | appendonly=true, orientation=column | orientation=column, compression=low | 需改写 |
| 压缩写法 | 随appendonly配置 | 独立compression参数 | 需改写 |
| 分布键 | distributed by (id) | distribute by hash(id) | **高频改动** |
| 随机分布 | distributed randomly | distribute by roundrobin | 需改写 |
| 分区表 | 仅start...end | less...than + start...end | DWS更灵活 |
| 字符长度 | 按字符计算 | 按字节计算（nvarchar2按字符） | **注意隐含截断** |
| 外部表 | external table | foreign table | 需改写 |
| CTAS分布键 | ... distributed by(xxx) 末尾 | ... distribute by hash(xxx) 在AS之后 | 需改写 |
| 游标定义 | cursor_name cursor for | cursor cursor_name for | 需改写 |

## 五、风险点识别

### 5.1 已识别风险

| 风险等级 | 风险项 | 说明 | 建议 |
|----------|--------|------|------|
| 高 | **Python UDF不支持** | GP上可能有Python UDF，迁移后将无法直接使用 | 提前排查GP中Python UDF使用情况，评估重写为PL/pgSQL或C UDF |
| 中 | **字符长度语义差异** | GP按字符计算，DWS默认按字节计算，中文场景可能截断 | 排查含中文字段的表，统一改用nvarchar2或扩大varchar长度 |
| 中 | **存储过程转换率99.37%** | 10个存储过程需人工处理 | 提前识别这10个存储过程的复杂度 |
| 中 | **BI工具兼容性** | Yonghong BI需JDBC调用方式 | 验证现有BI报表工具的兼容性 |
| 低 | **实时SQL监控** | 本地版需后台查看视图 | 云上版本已支持，评估是否可接受 |
| 低 | **DDL源码查看** | 部分DDL建议用DataStudio | 习惯改变，影响小 |

## 八、总结评价

### 8.1 综合评分

| 评估维度 | 评分(1-10) | 说明 |
|----------|-----------|------|
| 功能完整性 | **9** | 基本功能全部通过，仅Python UDF缺失 |
| 迁移能力 | **9** | DDL 100%，存储过程 99.37%，数据迁移速度快 |
| 性能表现 | **9** | TPC-H 7min50s稳定，并发线性扩展，列存压缩优秀 |
| 高可用性 | **9** | CN多活4s切换，DN 60s自动切换 |
| 兼容性 | **8** | GP语法90%+兼容，字符长度语义需注意 |
| 运维管理 | **7** | 监控工具基本完善，实时SQL监控本地版需改进 |
| **综合** | **8.5** | **满足MPP选型需求，推荐进入下一阶段详细POC** |

### 8.2 核心结论

1. **DWS与GP同属PG体系**，语法相似度90%+，迁移路径清晰
2. **自动迁移工具成熟**，DDL转换100%，大幅降低人工成本
3. **性能表现优秀**，TPC-H稳定、并发扩展良好、列存压缩7:1
4. **高可用架构完备**，CN多活+DN主备从三级冗余
5. **主要风险可控**：Python UDF需提前排查、字符长度语义需统一处理
6. **建议**：进入详细POC阶段，重点验证生产环境全量迁移和业务SQL兼容性
